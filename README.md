## Project Overview

An end-to-end AI analytics platform built on **25,000 hospital visit records** across **5,000 patients** in a multi-city hospital network. The platform delivers two machine learning classification systems, a real-time prediction API, and a structured SQL analytics layer — addressing three critical hospital leadership challenges:

| Challenge | Solution |
|---|---|
| Patient flow inefficiency | Model A — Visit Risk Classifier |
| Revenue leakage from claim rejections | Model B — Claim Outcome Classifier |
| Delayed payment cycles | SQL financial analytics + API pre-submission flagging |

**Estimated Annual ROI: ₹4.0–4.9 crore**

## Standalone Scripts

Two utility scripts are provided at the project root for command-line use:

### `build_features.py` — Feature Engineering
Extracts and engineers all 14 features from `hospital.db` and saves `model_table.csv` and `feature_schema.json` to `Data_Outputs/`. Run this before Phase 3 if you want to regenerate features without opening the notebook.

```bash
python build_features.py                          # default paths
python build_features.py --db ./hospital.db       # custom DB path
python build_features.py --out ./Data_Outputs     # custom output path
```

### `monitor.py` — Drift Monitoring
Runs PSI-based feature drift detection, input validation, and generates a retrain decision report. Run this monthly in production or after any data pipeline change.

```bash
python monitor.py                                          # train vs test drift check
python monitor.py --new-data Data_Outputs/new_data_sample.csv  # check against new data
python monitor.py --report-only                            # print last report only
```

## How to Run

### Prerequisites
```bash
pip install pandas numpy matplotlib seaborn scikit-learn joblib \
            fastapi uvicorn pydantic
```

### Step 1 — Set up the database
Open and run **`Notebooks/Phase1_SQL_Analytics.ipynb`**

Creates `hospital.db` (SQLite) from the three raw CSV files and runs all operational and financial SQL queries.

> ⚠️ Phase 1 must be run first — all other notebooks depend on `hospital.db`

### Step 2 — EDA & Feature Engineering
Run **`Notebooks/Phase2_EDA_DataQuality.ipynb`**

Or use the standalone script:
```bash
python build_features.py
```

### Step 3a — Model A: Visit Risk Classification
Run **`Notebooks/Phase3_02_risk_model.ipynb`**

Trains a Random Forest classifier (with GridSearchCV tuning) to predict visit risk:
- **Target:** `risk_score` — Low / Medium / High
- **Saves:** `phase3_models/artifacts/modelA_rf.joblib`

### Step 3b — Model B: Claim Outcome Classification
Run **`Notebooks/Phase3_03_claim_model.ipynb`**

Trains a Random Forest classifier (with GridSearchCV tuning) to predict claim outcome:
- **Target:** `claim_status` — Paid / Pending / Rejected
- **Saves:** `phase3_models/artifacts/modelB_rf.joblib`

### Step 4 — Evaluation & Explainability
Run **`Notebooks/Phase4_Evaluation_Explainability.ipynb`**

Generates confusion matrices, precision/recall/F1, feature importance, fairness segmentation, and model card.

### Step 5 — API Deployment
Run **`Notebooks/Phase5_API_Deployment.ipynb`**

To start the API locally:
```bash
cd API/
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```
Visit `http://localhost:8000/docs` for interactive API documentation.

**Or via Docker (from project root):**
```bash
docker build -t hospital-risk-api:1.0.0 API/
docker run -d -p 8000:8000 \
  -v $(pwd)/Data_Outputs:/app/models \
  -v $(pwd)/API/logs:/app/logs \
  hospital-risk-api:1.0.0
```

### Step 6 — Monitoring & Governance
Run **`Notebooks/Phase6_Monitoring_Governance.ipynb`**

Or use the standalone script:
```bash
python monitor.py
```

## Governance Summary

- **Decision-support only** — no clinical or financial decision may be made solely on AI output
- **Retraining trigger** — PSI > 0.20 on any feature OR recall drop below 0.65
- **Quarterly retraining** on rolling 12-month data window
- **Fairness audits** — quarterly segmentation by gender, city, and insurance provider
- **All predictions logged** with timestamp, model version, and input hash
- Full details: `Governance/GOVERNANCE_COMPLIANCE.md`  
  Includes: Section 9 — full incident response procedures (P1/P2/P3 classification, escalation path, rollback procedure)
